# TrotBot 本地运行指南（配合官方 README）

本文档面向：**已将 `trotbot` 包放入 colcon 工作空间**（例如 `~/trotbot_ws/src/trotbot`），希望先跑通软件栈，再改 HAL 对接自有机器狗。

**本指南已在以下环境按步骤验证：** Ubuntu 22.04 arm64（RK358 等）、ROS 2 Humble、工作空间 `~/trotbot_ws`。

---

## 1. 版本与系统怎么选

| 说明 | 建议 |
|------|------|
| 官方 TrotBot README 徽章 | 写的是 **ROS 2 Jazzy**（常见于 Ubuntu 24.04） |
| **CHAMP** `ros2` 分支自述 | 明确写的是 **Ubuntu 22.04 + ROS 2 Humble** 通过测试 |
| 你当前若是 **Ubuntu 22.04** | **优先安装 ROS 2 Humble**，与 CHAMP 一致，踩坑最少 |
| 你若是 **Ubuntu 24.04** | 可安装 **Jazzy**，需自行 `colcon build` 验证；若依赖报错，再考虑 Docker 内用 22.04+Humble |

下文以 **ROS 2 Humble + Ubuntu 22.04** 为例；其他发行版请将命令中的 `humble` 换成你的 distro（如 `jazzy`）。

---

## 2. 安装 ROS 2、构建工具与 rosdep 源

1. 按官方文档安装 ROS 2 Humble：  
   [https://docs.ros.org/en/humble/Installation.html](https://docs.ros.org/en/humble/Installation.html)

2. 安装常用工具：

```bash
sudo apt update
sudo apt install -y python3-colcon-common-extensions python3-rosdep git
sudo rosdep init   # 若已初始化可跳过
```

3. **rosdep 换源（国内强烈建议）**  
   若 `rosdep update` 访问 `raw.githubusercontent.com` 超时，需要同时：
   - 在 `~/.bashrc` 中设置（或每次手动 export）：
     ```bash
     export ROSDISTRO_INDEX_URL=https://mirrors.tuna.tsinghua.edu.cn/rosdistro/index-v4.yaml
     ```
   - 将 `/etc/ros/rosdep/sources.list.d/20-default.list` 中的地址改为清华镜像（与本仓库同级的 `~/deep-dog/rosdep-mirror/` 下提供了列表副本与一键脚本 `install.sh`）。  
   然后再执行（**不要用 sudo**）：
   ```bash
   rosdep update
   ```

4. 每个新终端使用 ROS 前执行（若未写入 `~/.bashrc`）：

```bash
source /opt/ros/humble/setup.bash
```

## 3. 工作空间布局（关键：CHAMP 不在本仓库内）

`trotbot` 包依赖 **`champ_base`、`champ_msgs`**，遥控还依赖 **`champ_teleop`**。这些来自 **CHAMP** 仓库的 **`ros2` 分支**，需要与 `trotbot` 放在**同一个** `colcon` 工作空间里编译。

推荐目录结构：

```text
~/trotbot_ws/
  src/
    champ/          # git clone -b ros2 --recursive
    champ_teleop/   # git clone -b ros2
    trotbot/        # 本仓库：直接放在 src 下（或按需软链）
```

### 3.1 获取 CHAMP（含子模块 libchamp）

```bash
mkdir -p ~/trotbot_ws/src
cd ~/trotbot_ws/src

git clone --branch ros2 --recursive https://github.com/chvmp/champ.git
git clone --branch ros2 https://github.com/chvmp/champ_teleop.git
```

（可选用 `--depth 1` 浅克隆以节省时间。）

### 3.2 放入本仓库 `trotbot`

推荐将本仓库**直接**置于 `~/trotbot_ws/src/trotbot`（`git clone` 到该路径，或将目录移入工作空间）。若需在多处共享同一份源码，也可改用软链指向你的主目录。

```bash
# 可选：软链（源码在其它固定目录时）
# ln -sf /path/to/trotbot ~/trotbot_ws/src/trotbot
```

### 3.3 对 `champ_teleop` 打补丁（必做，与 Humble / TrotBot 对接）

官方 `champ_teleop` 的 `ros2` 分支存在：**`joy_node` 仍使用已废弃的 `dev` 参数**、与 **`dev_port` / `LaunchConfiguration` 不一致**等问题，在 ROS 2 Humble 下手柄常无法工作。

本仓库提供**已合并的 launch 补丁**（含 `device_id`、`use_xterm` 可选分支等）。在 clone 完 `champ_teleop` 后执行：

```bash
bash ~/trotbot_ws/src/trotbot/patches/apply_champ_teleop.sh ~/trotbot_ws/src
cd ~/trotbot_ws
source /opt/ros/humble/setup.bash
colcon build --symlink-install --packages-select champ_teleop
```

补丁源文件位于：`trotbot/patches/champ_teleop/teleop.launch.py`（可自行 `diff` 对照上游）。

## 4. 依赖与编译

```bash
cd ~/trotbot_ws
source /opt/ros/humble/setup.bash
export ROSDISTRO_INDEX_URL=https://mirrors.tuna.tsinghua.edu.cn/rosdistro/index-v4.yaml   # 若已写入 bashrc 可省略

rosdep install --from-paths src --ignore-src -r -y
```

说明：

- `rosdep` 会尝试安装 **Gazebo、Nav2** 等（来自 `champ_gazebo`、`champ_navigation`）。在 **arm64 + 定制系统库**（如 Rockchip 多媒体包替换 `libav*`）上，**`ros-humble-desktop` 或部分仿真依赖可能 apt 冲突**；若仅需 **真机 / RViz + 步态 + 遥控**，不必强行装全量仿真栈。
- **推荐最小可运行集合**（已在本机验证通过）：

```bash
colcon build --symlink-install --packages-select champ_msgs champ champ_base champ_teleop trotbot
source ~/trotbot_ws/install/setup.bash
```

- **`python3-scipy`**：`trotbot` 运行时需要。优先：
  ```bash
  sudo apt install -y python3-scipy
  ```
  若暂时无法使用 apt（例如 **`dpkg` 被其他 apt 进程占用**），可临时：
  ```bash
  pip3 install --user scipy
  ```
  并在 `~/.bashrc` 中加入（或每次运行前 export）：
  ```bash
  export PYTHONPATH="${HOME}/.local/lib/python3.10/site-packages:${PYTHONPATH:-}"
  ```
  长期仍建议改为 **apt 安装 `python3-scipy`**，与系统包管理一致。

- 本仓库已在 `rviz/`、`worlds/` 下放置占位，避免 `CMakeLists.txt` 中 `install(DIRECTORY ...)` 因目录缺失导致安装阶段失败。

## 5. 运行顺序（与官方 Quick Start 一致）

先编译并 source：

```bash
cd ~/trotbot_ws
source /opt/ros/humble/setup.bash
source install/setup.bash
# 若 scipy 来自 pip --user，请保留 PYTHONPATH（见上一节）
```

**终端 1 — 核心：步态控制 + 状态估计 + 舵机接口**

```bash
ros2 launch trotbot trotbot_basic.launch.py
```

**无 PCA9685 / 无 sysfs PWM（仅有线调试、不想看刷屏）**：关闭 `servo_interface` 节点，控制器仍会运行，只是关节指令不会下发到硬件：

```bash
ros2 launch trotbot trotbot_basic.launch.py use_servo_interface:=false
```

**同时打开 RViz（需 `DISPLAY`、已安装 `ros-humble-rviz2`）**：会额外启动 **`robot_state_publisher`**（用 `xacro` 展开 `description_file`，默认 **`minidog_champ.urdf.xacro`**）和 **`rviz2`**，并默认加载 **`share/trotbot/rviz/trotbot.rviz`**（可用 `rviz_config:=路径` 覆盖）。

```bash
ros2 launch trotbot trotbot_basic.launch.py use_servo_interface:=false rviz:=true
```

**RobotModel 应用「话题」还是「文件」**：与 Trotbot 配套时请用 **Description Source = Topic**，话题 **`/robot_description`**。`robot_state_publisher` 会发布该话题（通常带 **Transient Local**）；若 RViz 里 QoS 选成 **Volatile** 且启动晚于首条消息，会看不到模型。仓库里的 **`rviz/trotbot.rviz`** 已把 RobotModel 的 **Durability** 设为 **Transient Local**。不要直接在 RViz 里选 **File** 打开 **`*.xacro`**：`package://trotbot/meshes/...` 需要 ROS 包路径解析，裸加载文件会 **找不到 mesh**；若必须用文件，需先用 `xacro` 生成 URDF 并处理 mesh 路径。

**终端 2 — 遥控**

- **桌面有图形、有 xterm（默认）**：键盘在独立 xterm 中，与官方行为一致。

```bash
ros2 launch trotbot trotbot_teleop.launch.py
```

- **无 DISPLAY / 不想用 xterm**：在当前终端跑键盘节点（需 **真实 TTY**，例如 `ssh -t user@host`，否则可能 `termios` 报错）：

```bash
ros2 launch trotbot trotbot_teleop.launch.py use_xterm:=false
```

- **手柄**（根据 `joystick_dev` 路径末尾 **`jsN`** 自动设置 `joy_node` 的 **`device_id`**）：

```bash
ros2 launch trotbot trotbot_teleop.launch.py use_joystick:=true
ros2 launch trotbot trotbot_teleop.launch.py use_joystick:=true joystick_dev:=/dev/input/js1
```

**可选终端 3 — RViz**

```bash
source ~/trotbot_ws/install/setup.bash
ros2 run rviz2 rviz2
```

（官方 README 中的 `trotbot.rviz` 若未提供，可在 RViz 里手动添加 RobotModel、TF 等。）

## 6. 无实体硬件时会发生什么

`trotbot_basic.launch.py` 默认会启动 **`servo_interface`**，内部会初始化 **PCA9685 / sysfs PWM** 等。无硬件时常见现象：

- 日志中大量 **`sysfs PWM write failed`**、`pwmchip0/pwmXX` 不存在 — **不是控制器崩溃**，而是没有对应 PWM 设备；只要 **`quadruped_controller_node`、`state_estimation_node` 已启动**，即可配合 RViz、`/cmd_vel` 做轻量验证。
- **若希望终端干净**：使用 **`use_servo_interface:=false`**（见第 5 节），则不再启动舵机节点、也不再刷 sysfs 错误。
- 可能出现 **`Calibration file not found: .../.config/trotbot/nvmem`** — 标定文件缺失告警，无硬件调试阶段可暂不关心。

有实物时：按官方 README 开启 I2C、安装 `i2c-tools`，并确认 `sudo i2cdetect -y 1` 能看到 PCA9685（常见地址 `0x40`）。

## 8. 对接自有机器狗：改哪里

目标：保持 **CHAMP 上层**（`quadruped_controller`、步态、`/cmd_vel`）不变，替换 **底层执行**。

| 层次 | 路径 / 说明 |
|------|-------------|
| 关节指令入口 | `hal/interfaces/servo_interface.py`：订阅 `joint_group_effort_controller/joint_trajectory`，将轨迹点转为关节角 |
| HAL 与 PWM/舵机 | `hal/hardware/hardware_interface.py`、`hal/hardware/adafruit_servo_controller.py`、sysfs 相关实现 |
| 标定与映射 | `hal/hardware/servo_calibration.py`、`hal/hardware/config.py`（腿长、PWM 通道、脉宽范围等） |
| 机器人模型与 CHAMP 配置 | 默认 `urdf/minidog_champ.urdf.xacro`、`config/champ/joints.yaml`、`links.yaml`、`gait_minidog.yaml`（旧款可改 `description_file` / `gait_config_file`） |

建议顺序：先在你机器狗的 URDF + `joints.yaml` / `links.yaml` / `gait.yaml` 上与 CHAMP 对齐关节命名与运动学，再实现 `HardwareInterface` 的“角度 → 你的执行器命令（CAN/串口/其他）”。

CHAMP 硬件集成思路可参考：  
[https://github.com/chvmp/champ/wiki/Hardware-Integration](https://github.com/chvmp/champ/wiki/Hardware-Integration)（以 ROS1 文档为主，ROS2 话题名略有差异，但分层相同）。

## 9. 与官方 README 的差异小结

- 官方写的 `~/TrotBot/trotbot_ws` 仅为示例路径，你可使用任意路径（如 `~/trotbot_ws`），关键是 **`src` 里同时有 `champ`（ros2 + recursive）、`champ_teleop`（ros2）、`trotbot`**。
- **必须**使用 CHAMP 的 **`ros2` 分支**，不要用 `master`（ROS1）。
- Ubuntu 22.04 建议 **Humble**；README 上的 Jazzy 更适合 24.04，需自行验证。
- 本仓库已补 **`rviz/`、`worlds/`** 空目录；**务必按第 3.3 节对 `champ_teleop` 打补丁**；`trotbot_teleop.launch.py` 会从 **`joystick_dev` 解析 `device_id`** 并支持 **`use_xterm`**。

## 10. 已知问题与本机实践记录

便于以后在同一类板子或网络环境下查阅。

| 现象 | 原因与处理 |
|------|------------|
| `rosdep update` 超时、仍访问 `raw.githubusercontent.com` | 仅设置 `ROSDISTRO_INDEX_URL` 不够，还需把 **`/etc/ros/rosdep/sources.list.d/20-default.list`** 改为镜像；见第 2 节与 `~/deep-dog/rosdep-mirror/`。 |
| `ros2 pkg list \| head` 报 `BrokenPipeError` | `head` 提前关闭管道导致，**非安装错误**；不要用管道或改用重定向文件再 `head`。 |
| 安装 `ros-humble-desktop` 时 **`libavcodec-dev` 等依赖冲突** | 常见於 **Rockchip 等 BSP** 替换过 `libav*`。可只装 **`ros-humble-ros-base`**，再按需 **`apt install ros-humble-rviz2`**；不必强装 desktop 元包。 |
| `rosdep install` / `apt` 报 **`dpkg` 锁被占用** | 等待其它 **`apt-get`/`apt`** 结束后再执行；期间可用 **`pip install --user scipy`** 临时满足 `trotbot`（见第 4 节）。 |
| `trotbot_basic` 中 **`servo_interface` 刷屏 sysfs 错误** | 无 PWM/PCA9685 设备时的**预期现象**；控制器节点正常即可继续调试。 |
| **`Calibration file not found: ... nvmem`** | 标定未写入 `~/.config/trotbot/`；有实物后再做标定流程。 |
| **无 DISPLAY 时 `xterm` 启动失败** | 使用 `use_xterm:=false` 并保证 **真实 TTY**，或仅用 **`/cmd_vel`**（第 5、7 节）。 |
| **IDE/自动化终端里 `use_xterm:=false` 仍 `termios` 报错** | 该环境**不是交互式 TTY**；在实体串口或 `ssh -t` 会话中运行，或改用 **`/cmd_vel`**。 |
| 需要 **Gazebo + Nav2 全栈** | 在依赖不冲突的机器上对工作空间**全量** `colcon build`，或单独解决 **`champ_gazebo` / `champ_navigation`** 的 apt 依赖；本指南的最小集**不包含**仿真与导航编译。 |
| **`rqt_graph` 里看不到「话题名」当一颗节点** | **正常**。ROS 图里一般是 **节点（椭圆）** 与 **节点之间的箭头/连线** 表示订阅关系；**话题名**常显示在连线上或需在界面里切换「显示话题」类选项。`/joint_group_effort_controller/joint_trajectory` 的**发布者**是 **`quadruped_controller_node`**（包 `champ_base`），**订阅者**是 **`servo_interface`**（若已启动）。 |

## 11. 关节轨迹话题、`rqt_graph` 与无刷 / 力矩扩展思路

本节汇总：**谁在发关节指令**、**消息里有什么**、**如何在终端里看**、以及日后 **MIT 运控 / 无刷** 若要 **速度、力矩** 时的大致改法（当前仓库**未实现**，仅作路线说明）。

### 11.1 `rqt_graph` 里话题怎么出现？

- **`rqt_graph` 画的是「节点」与「通信关系」**，**不会**把每个话题都画成一颗独立椭圆节点（除非使用某些「按话题分组」的视图或插件）。
- 典型看法：找到 **`quadruped_controller_node`**，看它**连出去**的线，对应话题可在连线上标注或通过 **`ros2 topic info`** 核对。
- 安装与启动：
  ```bash
  sudo apt install -y ros-humble-rqt-graph   # 若未装
  rqt_graph
  # 或
  ros2 run rqt_graph rqt_graph
  ```

### 11.2 `/joint_group_effort_controller/joint_trajectory` 从哪里来？

| 项目 | 说明 |
|------|------|
| **发布节点** | **`quadruped_controller_node`**（`champ_base` / CHAMP 四足控制器） |
| **话题名为何带 effort** | 来自 `launch/champ_controllers.launch.py` 里参数 **`joint_controller_topic`**，默认写成 `joint_group_effort_controller/joint_trajectory`，**名字是历史习惯**；**不表示**消息里一定有力矩。 |
| **订阅节点** | **`servo_interface`**（`trotbot` 包）：收到后只使用 **`positions`**，经 HAL 转成舵机 PWM。 |

### 11.3 消息类型与实际字段

- **类型**：`trajectory_msgs/msg/JointTrajectory`
- **当前 CHAMP 实现**（`champ_base` 中 `publishJoints_`）：为每个控制周期构造 **一个** `JointTrajectoryPoint`，**只填充 `positions`（12 个关节，单位弧度）**；**`velocities`、`accelerations`、`effort` 为空数组**。
- **`time_from_start`**：代码里设为约 **1/60 s** 量级，表示轨迹点的时间语义；**实际控制周期**由控制器参数 **`loop_rate`** 决定（源码默认 **200 Hz**，即约每 5 ms 一次循环并发布，若你未在参数里覆盖）。

在终端查看实时消息：

```bash
ros2 topic echo /joint_group_effort_controller/joint_trajectory
ros2 topic info /joint_group_effort_controller/joint_trajectory
ros2 interface show trajectory_msgs/msg/JointTrajectory
```

### 11.4 若要做无刷 + MIT 类运控（速度 / 力矩），原则上改哪里？

当前链路是 **位置开环到舵机**；**MIT 风格**常需要 **关节侧速度、前馈力矩** 等，**不能**指望只改 `servo_interface` 就从空数组里「变出」物理意义正确的速度/力矩，需要在 **规划或控制层** 生成这些量。

可选方向（可并行评估，不必只选一条）：

1. **改 CHAMP 发布侧**  
   在 **`~/trotbot_ws/src/champ/champ_base/src/quadruped_controller.cpp`** 的 **`publishJoints_`** 中，除 `positions` 外，按你的控制律填写 **`JointTrajectoryPoint.velocities` / `effort`**（需有 **角速度估计** 与 **力矩模型或 PD**）。改完需 **重新 `colcon build champ_base`**，维护成本随 CHAMP 升级而上升。

2. **中间适配节点（推荐与上游解耦时）**  
   保留 CHAMP 只发 **位置**；新建节点订阅该话题（或 `/joint_states`），在内部做 **MIT 控制律 / 通信打包**，再通过 **自定义话题或 CAN** 发给电机驱动。`servo_interface` 可整体替换为订阅新话题的节点。

3. **`ros2_control` + 自定义硬件接口**  
   用 **Hardware Interface** 对接无刷驱动器，在 **控制器插件** 里输出力矩/电流指令；与 CHAMP 并存或替换需要 **架构设计**（话题与控制器管理器配置）。

4. **与 CHAMP 解耦的独立运控栈**  
   若 MIT 控制器在 **MCU** 上闭环，上位机只发 **高层指令**，则可能 **不再使用** `JointTrajectory` 这一路，而改用你们自定义的 **低层协议**。

**小结**：要看 **`/joint_group_effort_controller/joint_trajectory` 的发布图**，请在 **`rqt_graph` 里找 `quadruped_controller_node`**；消息 **当前仅有位置**；无刷 + MIT 需在 **CHAMP 内补全轨迹消息** 或 **另建控制/适配层**，而不是仅靠现有空字段自动生效。

### 11.5 Mini Cheetah 电机五元组（p、v、τ、kp、kd）与开源生态（调研备忘）

**CHAMP / Trotbot 能否直接输出这五元组？**  
**不能。** 当前只稳定输出 **关节目标位置 `p`**（弧度）。**`v`（目标角速度）、`τ`（前馈力矩）** 需你们用 **数值微分、雅可比×足端速度、重力/动力学补偿、简化 PD 残差** 等方式在 **CHAMP 之外或改 CHAMP** 生成；**`kp`、`kd`** 一般由 **按关节/按步态相位的查表、固定标定、或上层 WBC** 给定，也不会由现成 Trotbot 自动给出。

**是否存在「官方 Mini Cheetah + ROS2」一键融合包？**  
**没有**与 CHAMP 直接对接的、现成的「MIT 整机软件栈 ROS2 元包」。MIT 侧代码多为 **研究用整体仓库**（Simulink/自研框架 + 嵌入式），不是 `apt install` 就能和 `champ_teleop` 拼在一起的插件。

**可复用的开源层次（需自写「胶水」节点或 `ros2_control` 接口）：**

| 层次 | 代表方向 | 说明 |
|------|----------|------|
| **CAN 协议 / 电机库** | [dfki-ric-underactuated-lab/mini-cheetah-tmotor-can](https://github.com/dfki-ric-underactuated-lab/mini-cheetah-tmotor-can)、[well-robotics/cubemars-tmotor-can](https://github.com/well-robotics/cubemars-tmotor-can) 等 | **C++/SocketCAN** 发 MIT 风格数据帧；通常 **不是**完整 ROS2 节点，需在你们工作空间里 **包一层 Node** 订阅自定义消息再调库。 |
| **硬件抽象** | **`ros2_control` 自定义 `SystemInterface`** | 把标准 **JointState / 力矩指令** 映射到上述 CAN 库；**控制器**仍要生成 p、v、τ 或等效量。 |
| **高层步态** | **CHAMP**（现状） | 继续出 **p**；**v、τ** 在适配节点里算，或 **Fork `quadruped_controller.cpp`** 在 `publishJoints_` 里扩展（维护成本高）。 |
| **整机 MPC/WBC** | Cheetah 3 / Mini 相关论文与闭源或重实现 | 与 CHAMP **不同路线**；若要做真 MIT 全身动力学，工作量接近 **新项目**，不是「融合一个功能包」级别。 |

**推荐集成思路（与当前仓库衔接）：**

1. **保留 CHAMP** 做步态与 **目标关节角 `p_ref`**。  
2. **新建 `mit_motor_bridge`（名称自定）节点**（或 MCU）：订阅 `p_ref`（或从 `JointTrajectory` 取 `positions`），结合 **上一周期 `p_ref` 差分或 URDF+运动学** 估 **`v_ref`**，**`τ_ff`** 可用 **重力补偿** 或先 **置 0** 调通链路；**`kp`、`kd`** 用参数文件按关节配置。  
3. 节点内调用 **mini-cheetah-tmotor-can / cubemars-tmotor-can** 同类库，经 **SocketCAN** 发帧；频率建议与驱动器文档一致（常 **≥500 Hz～1 kHz**），必要时 **PREEMPT_RT** 或 **下沉 MCU**。  
4. **不要指望**把 Mini Cheetah **官方仓库**当 ROS2 依赖直接 `find_package`；以 **协议文档 + CAN 库 + 自写桥接** 为主。

**结论**：**p** 可由当前项目给出；**v、τ、kp、kd** 需 **自研适配层或改 CHAMP**；开源里 **现成可复用的是 CAN/电机驱动库与 `ros2_control` 模式**，**没有**与 Trotbot 零配置的「Mini Cheetah 功能包融合」方案。

### 11.6 本机四足底层 CAN 开发路线（RK3588 双路 CAN + 灵足电机 / 类 MIT 协议）— 评估与备忘

以下是对「**参考 DFKI / well-robotics 两库做底层驱动、RK3588 双 CAN 分前后腿、电机为灵足时代且协议近似小米/MIT**」的 **可行性评估** 与 **下一步开发要点**，便于与 CHAMP 只发 **位置** 的现状衔接。

#### 是否可以把两库当「写机器狗 CAN 底层」的参考？

**可以，定位是「协议与编码思路参考」，不是直接拷贝即兼容灵足。**

- 两库解决的是：**在 Linux SocketCAN 上，把 float 域（p、v、kp、kd、τ）按 **Ben Katz / CubeMars AK 系** 常见方式 **量化、组 8 字节 CAN 帧、收发与解析反馈**（位置、速度、力矩等）。  
- 你当前栈里 CHAMP 只给 **`p`（关节角）**：可在 **自研桥接节点** 内 **补 `v`、`τ_ff`**（差分、模型或先置 0）、**配置 `kp`/`kd`**（yaml），再 **调用与两库同构的组帧逻辑**（若灵足帧格式一致）或 **按灵足手册实现 `encode`/`decode`**。  
- **反馈解析**：两库的 **`decodeCANFrame` / `motorState` 模式**可直接借鉴；最终 **必须以灵足官方协议为准** 实现或校对。

#### RK3588 两路 CAN：前腿 / 后腿分总线

**架构上完全可行**，也是常见做法（降低单总线负载、隔离故障）。

- **硬件**：确认板卡 **两路 CAN 控制器** 在 Linux 中是否为 **`can0`、`can1`**（或 `can2` 等），分别 **配置 bitrate**（多与驱动器一致，如 1 Mbps，**以灵足文档为准**）。  
- **软件**：桥接节点内维护 **「关节索引 → (接口名, CAN ID)」** 映射，例如前腿 6 个电机走 **`can0`**、后腿 6 个走 **`can1`**；或使用 **两个 `MotorDriver` 实例** / 两套 socket，分别绑定不同 `motor_can_socket`。  
- **注意**：**终端电阻**、**地线共地**、**ID 规划**（全局唯一）要与布线一致，避免两路接错设备。

#### 灵足时代电机「与小米电机 / MIT 狗子协议高度相似」— 风险说明

- **相似**通常指：**也是「多 float 域压缩进 64 bit」一类准直驱协议**，开发上可以 **对照两库的 `float_to_uint` / `encodeCANFrame` 流程** 快速搭原型。  
- **不能默认 bit 级兼容**：不同厂商可能在 **字节序、域宽、命令字、扩展帧/标准帧、使能/清零指令** 上与 AK 系 **不一致**。**上线前必须用示波器/逻辑分析仪或 `candump` 与官方文档逐项核对**，必要时 **单独写 `lingzu_encode`/`lingzu_decode`**，仅把两库当 **参考实现**。

#### 与当前 Trotbot 的衔接方式（建议）

| 步骤 | 内容 |
|------|------|
| 1 | **冻结** CHAMP 输出：`/joint_group_effort_controller/joint_trajectory` 的 **`positions`** + `joint_names` 顺序。 |
| 2 | **新建** 包（如 `trotbot_can_bridge`）：订阅上述话题（或等价自定义话题），**不再使用** `servo_interface` 的 PWM 路径（真机无刷时）。 |
| 3 | 在桥接内：**组五元组**（`p` 来自 CHAMP；`v`、`τ`、`kp`、`kd` 按策略填）→ **组 CAN 帧** → **`can0`/`can1` 分发**。 |
| 4 | **读回** 反馈帧 → 发布 **`/joint_states`**（或仅内部用于闭环），供 `robot_state_publisher` / 记录使用。 |
| 5 | **实时性**：RK3588 上尽量 **固定周期线程 + SCHED_FIFO** 或 **MCU 专跑 1 kHz CAN**；两库 README 亦提到非实时系统 `usleep` 不准的问题。 |

#### 下一步开发清单（可打勾推进）

1. 获取 **灵足时代电机官方 CAN 协议 PDF**（命令帧、反馈帧、波特率、ID 规则）。  
2. 用 **`candump`** 与 **单电机台架** 验证与 **AK/MIT 示例帧** 是否一致；不一致则 **独立实现编解码**。  
3. 画 **关节 ↔ CAN 接口 ↔ CAN ID** 表（12 电机）。  
4. 实现 **最小桥接**：仅 `p` + 固定 `kp,kd` + `v=0,τ=0` 上电试转。  
5. 再迭代 **速度前馈、力矩前馈** 与 **安全限幅/急停**。  

**总结**：两开源库 **适合作为 RK3588 上 SocketCAN 底层驱动的参考实现**；**双 CAN 分前后腿合理**；**位置-only 的 CHAMP 输出需经自研桥接补全五元组再发 CAN**；**灵足协议务必以厂商文档为准做最终核对**，勿默认与小米/MIT 帧逐位相同。

### 11.7 LubanCat-4-V1 双 CAN 落地要点（结合官方手册）

### 11.8 最小桥接实现（`/joint_trajectory` -> 双 CAN）

按当前仓库分层，建议把 CAN 桥接做成一个独立 ROS2 包（例如 `trotbot_can_bridge`），角色对齐现有 HAL：

- 上游输入：订阅 `/joint_group_effort_controller/joint_trajectory`
- 中间转换：按关节名取 `positions`，做 **URDF角度 -> 电机目标角** 的零位换算
- 下游输出：按关节映射表发到 `can0/can1` 对应电机 ID
- 反馈闭环：解析反馈帧，发布 `/joint_states`（或内部状态）

**最小可跑版本（你提的方案）**：

1. 仅使用 `p_ref`（来自 `JointTrajectory.positions`）  
2. 固定参数：`kp=Kp_const`、`kd=Kd_const`  
3. 置零：`v_ref=0`、`tau_ff=0`  
4. 组帧发送到电机  
5. 用 `candump can0`、`candump can1` 同时抓包确认报文频率与字段

建议新增配置文件（YAML）管理这些静态信息：

| 配置项 | 示例 |
|------|------|
| `joint_to_bus` | `lf_hip: can0`, `rf_hip: can0`, `lh_hip: can1`, `rh_hip: can1` |
| `joint_to_motor_id` | `lf_hip: 0x01` ... |
| `joint_sign` | `+1/-1`（URDF正向与电机正向一致性） |
| `joint_zero_offset_rad` | 每个关节的零位偏置（弧度） |
| `kp/kd` | 可先全局固定，再细化到每关节 |

### 11.9 URDF 零位与电机零位不一致：怎么定？

你的判断是对的：**必须做零位转换**，不能直接把 URDF 角度当电机角度发。

建议采用统一公式（每个关节一套参数）：

```text
motor_pos_rad = sign * (urdf_pos_rad - urdf_zero_rad) + motor_zero_rad + offset_rad
```

多数项目可简化为：

```text
motor_pos_rad = sign * urdf_pos_rad + joint_zero_offset_rad
```

其中 `joint_zero_offset_rad` 通过一次标定得到。

**零位标定姿态建议**（优先顺序）：

1. **优先：机械中位附近、远离限位的“站立标定姿态”**（推荐）  
2. 不建议把**机械极限位置**作为零位（风险：接近硬限位、误差放大、容错差）  
3. “狗子放平”是否可用：可以作为一致性姿态，但要满足各关节都不靠近限位、且可重复摆放

一句话原则：**零位要选“可重复 + 安全 + 远离限位 + 与控制常用工况接近”的姿态**，通常就是中立站姿或吊装中立姿态。

### 11.10 若让我帮你完成角度换算，需要你提供哪些数据？

请按每个关节（12个）提供下表信息：

| 数据 | 说明 |
|------|------|
| 关节名 | 与 `/joint_trajectory.joint_names` 一一对应 |
| 电机 ID + 总线 | 例如 `lf_hip -> can0, 0x01` |
| 方向 `sign` | URDF 正方向与电机正方向同向填 `+1`，反向填 `-1` |
| 电机零位定义 | 电机协议里的“0位”含义（上电绝对位/指令清零位） |
| 标定姿态下的电机实测角 | 建议用厂家工具或反馈帧读出（弧度或可换算单位） |
| 标定姿态下的 URDF 目标角 | 同一时刻对应的 URDF 角（通常中立姿态） |
| 机械限位 | 最小/最大安全角（URDF侧和电机侧都尽量给） |
| 传动比/减速比 | 若反馈是电机轴角而非关节角，必须提供 |

拿到这些数据后，就能产出：

- 每关节 `joint_zero_offset_rad`
- 每关节安全限幅（防撞限位）
- `/joint_trajectory -> 电机角` 的可执行转换代码与参数文件

### 11.13 CAN 接口异常时的快速判断

若启动 bridge 提示 `can0/can1` 不存在，请先做下面检查：

```bash
ip -br link
sudo ip -details link show can0
sudo ip -details link show can1

# 检查实时设备树是否已启用 CAN 控制器
od -An -tc /proc/device-tree/can@fea50000/status
od -An -tc /proc/device-tree/can@fea70000/status
```

判定规则：

- 若 `ip -br link` 无 `can0/can1` 且 `status` 显示 `disabled`，说明本次启动未把 CAN overlay 应用到实时设备树，属于系统启动态问题，不是桥接节点问题。
- 这时即使 `ubuntuEnv.txt` 写了 overlay，也需要先解决“启动时 overlay 未生效”问题，再进行 CAN 发收测试。

### 11.14 运行指南与开发计划是否要分开？

建议 **分开**，原因：

- `本地运行指南` 适合放“可直接执行”的稳定步骤（安装、启动、排障）

- `本地运行指南.md`：保留“如何跑起来 + 当前推荐架构 + 最小桥接入口”
