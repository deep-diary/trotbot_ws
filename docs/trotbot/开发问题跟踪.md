### ISSUE-0001 关闭 feedback_step_limit 后动作仍明显滞后

- 发现时间：2026-04-27
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`，`control_gains.yaml`，launch 参数加载链路
- 优先级：P0 / P1 / P2
- 当前状态：In Progress

#### 1) 现象

- 已将 `enable_feedback_step_limit: false`，但机器人对姿态/行走指令响应仍明显滞后，体感类似阈值步进仍在生效。

#### 2) 复现步骤

1. 修改 `src/trotbot_can_bridge/config/control_gains.yaml`：`enable_feedback_step_limit: false`。
2. 重启 launch（未确认是否重建安装包、是否显式指定 `gains_file`）。
3. 下发较大姿态变化（如前倾）或行走速度变化，观察跟随性。

#### 3) 期望与实际

- 期望：关闭该开关后，轨迹目标应不再被“基于反馈阈值步进”截断。
- 实际：仍出现显著慢响应/滞后。

#### 4) 影响评估

- 影响姿态调整与步态跟随，表现为控制带宽不足，指令与机体动作不同步。

#### 5) 临时绕过

- 临时提高 `command_max_velocity_rad_s`（当前配置为 `1.0`，可能过低）。
- 临时关闭 `enable_startup_smoothing` 与 `enable_mode_rising_smoothing` 做 A/B 对比。
- 启动时显式指定 `gains_file:=<workspace>/src/trotbot_can_bridge/config/control_gains.yaml` 避免读取旧安装副本。

#### 6) 根因分析

- `enable_feedback_step_limit` 只控制 `SmoothJointCommand()` 的一段分支逻辑；即使关闭，仍会执行速度限幅：
  - `max_step = command_max_velocity_rad_s * dt`
  - `delta = clamp(target - prev, -max_step, +max_step)`
- 当前配置 `command_max_velocity_rad_s: 1.0`，若控制周期约 20ms，则单周期仅约 0.02rad，较大目标变化会表现为明显“慢慢追”。
- 当前配置还开启了 `enable_startup_smoothing: true`（3.0s）和 `enable_mode_rising_smoothing: true`，在上电阶段/模式切换后会进一步平滑目标。
- launch 默认 `gains_file` 来自 `FindPackageShare("trotbot_can_bridge")/config/control_gains.yaml`，若非 `--symlink-install` 或未重建，可能读取 install 目录中的旧 YAML，而非 src 中最新修改。

#### 7) 修复与验证

- 修复内容：
  - 核验运行态参数实际值（`enable_feedback_step_limit`、`command_max_velocity_rad_s`、`enable_startup_smoothing`、`startup_smoothing_duration_s`）。
  - 统一参数生效链路：显式 `gains_file` 指向工作区 src 配置，或重建安装后再启动。
  - 根据实机跟随性调大 `command_max_velocity_rad_s`（建议先试 6~12 rad/s 区间）。
- 验证命令/方法：
  - `ros2 param get /motor_protocol_node enable_feedback_step_limit`
  - `ros2 param get /motor_protocol_node command_max_velocity_rad_s`
  - `ros2 param get /motor_protocol_node enable_startup_smoothing`
  - `ros2 launch trotbot_can_bridge can_bridge.launch.py gains_file:=<abs_path_to_src>/control_gains.yaml`
- 验证结果：
  - 待你现场按上述步骤复测后回填。

#### 8) 结论

- 初步判断为“多重平滑/限幅叠加 + 参数文件来源不确定”导致的表象，不是单一 `feedback_step_limit` 开关失效。

### ISSUE-0002 前腿卡顿与前后腿发送频率长期失衡

- 发现时间：2026-04-27（夜间联调）
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`、`trotbot_can_bridge/can_transport_node`、`trotbot/config/champ/gait_minidog.yaml`、CAN overlay
- 优先级：P0 / P1 / P2
- 当前状态：In Progress

#### 1) 现象

- 悬空跑时前腿明显“卡顿/顿挫”，后腿相对平滑。
- 实测前后腿发包频率长期不均衡，前腿显著低于后腿。
- 单独屏蔽后腿仅发前腿时，前腿观感明显改善。

#### 2) 复现步骤

1. 正常启动 launch，双总线都开启（`tx_enable_can0=true`，`tx_enable_can1=true`）。
2. 手柄持续前进，抓取 `/can_frames_tx_line` 与 `/rosout`（`CAN bus counts` / `TX enqueue bus split`）。
3. 统计每电机 10s 发送频率与 can0/can1 总线发包比。

#### 3) 期望与实际

- 期望：双总线打开时，前后腿发包总量接近 1:1；每电机频率接近同一量级。
- 实际：多轮测到 `can0:can1` 约 `1:2`，并出现个别电机极低频（早期常见 11/12/13，后续漂移到 22/23/51）。

#### 4) 影响评估

- 直接影响前腿动作连续性与步态跟随观感，导致“卡顿”。
- 在动态阶段造成关节更新不一致，放大姿态抖动和相位误差。

#### 5) 临时绕过

- 临时仅发前腿（`tx_enable_can0=true`，`tx_enable_can1=false`）可显著改善前腿观感。
- 临时仅发后腿可复现实验对照（确认问题并非单腿硬件缺陷）。

#### 6) 根因分析（当前结论）

- 不是单纯 CAN 链路错误：`TX bus health(5s)` 长时间 `fail/retry/drop=0`。
- 不是 overlay 缺失：已确认 CAN 插件正常加载，接口稳定 up。
- 核心问题是“发布侧到发送侧存在分布性失衡”，表现为不同电机命中频率长期不一致。
- `/joint_group_effort_controller/joint_trajectory` 维度完整（12关节），但下游每电机发包仍出现漂移型低频点，说明仅靠参数调节难以彻底消除。

#### 7) 已做修复/实验（按时间顺序）

- 增加观测：
  - `Command clamp stats(total)`（CHAMP/MIT限位裁剪率）
  - `TX bus health(5s)`（按总线 fail/retry/drop）
  - `TX enqueue bus split(5s)`（按总线入队分布）
- `SmoothJointCommand` 修复：
  - 从全局 `dt` 改为每关节独立 `dt`，显著改善“命令被压扁”问题。
- 启动平滑优化：
  - 增加 `limit_velocity_only_during_smoothing`，避免稳态持续压幅。
- 临时总线开关：
  - 新增 `tx_enable_can0/tx_enable_can1`，用于前后腿独立 A/B。
- 发布路径调整：
  - `OnTrajectory` 改为固定索引映射全覆盖发送（并对缺失位使用上次目标兜底）。
- 发送频率约束：
  - 新增 `enable_min_tx_refresh/min_tx_refresh_interval_s`
  - 新增 `enable_max_tx_rate_limit/max_tx_rate_per_motor_hz`
  - 结论：可改变分布形态，但尚未彻底消除“漂移型低频电机”。
- CAN 时钟核对与对齐官方建议：
  - 实机比特率为 1M；
  - overlay `assigned-clock-rates` 已由 200MHz 改为 300MHz（官方 1M 推荐值），重启后生效。

#### 8) 关键实测结论（摘录）

- 单前腿发送时：总线约 `560Hz`，观感明显改善。
- 单后腿发送时：总线约 `560Hz`，形态与单前腿镜像。
- 双腿同时发送时：多轮出现 `can0` 明显低于 `can1`，且个别电机频率异常低。
- 最近一次（CAN时钟300M + 双腿开启）虽总量接近 1:1，但仍有个别电机掉频（如 23、51），问题未彻底闭环。

#### 9) 下一步计划（下次联调入口）

- 继续做“每电机发送原因计数”诊断（按电机统计 skip 原因）：
  - bus 开关跳过
  - 最大频率限制跳过
  - 最小刷新补发命中
  - 真实发送成功计数
- 在不影响动作的前提下，先锁死“每周期每电机必发”路径做对照基线，再逐步恢复优化器（平滑/刷新/限速）找最小可用集合。

#### 10) 结论

- 本问题已从“体感异常”收敛到“可量化的每电机发送分布异常”；
- 主要矛盾已定位在下游发布/发送链路的频率分配一致性，不是单纯 CAN 比特率或硬件链路故障；
- 需下一轮按“每电机原因计数”继续收尾。
