### ISSUE-0001 关闭 feedback_step_limit 后动作仍明显滞后

- 发现时间：2026-04-27
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`，`control_gains.yaml`，launch 参数加载链路
- 优先级：P0 / P1 / P2
- 当前状态：Resolved（待实机步态体感回归后转 Closed）

#### 1) 现象

- 已将 `enable_feedback_step_limit: false`，但机器人对姿态/行走指令响应仍明显滞后，体感类似阈值步进仍在生效。

#### 2) 复现步骤

1. 修改 `src/trotbot_can_bridge/config/control_gains.yaml`：`enable_feedback_step_limit: false`。
2. 重启 launch（未确认是否重建安装包、是否显式指定 `gains_file`）。
3. 下发较大姿态变化（如前倾）或行走速度变化，观察跟随性。

#### 3) 期望与实际

- 期望：关闭该开关后，轨迹目标应不再被“基于反馈阈值步进”截断。
- 实际：仍出现显著慢响应/滞后。

#### 4) 影响评估

- 影响姿态调整与步态跟随，表现为控制带宽不足，指令与机体动作不同步。

#### 5) 临时绕过

- 临时提高 `command_max_velocity_rad_s`（当前配置为 `1.0`，可能过低）。
- 临时关闭 `enable_startup_smoothing` 与 `enable_mode_rising_smoothing` 做 A/B 对比。
- 启动时显式指定 `gains_file:=<workspace>/src/trotbot_can_bridge/config/control_gains.yaml` 避免读取旧安装副本。

#### 6) 根因分析

- `enable_feedback_step_limit` 只控制 `SmoothJointCommand()` 的一段分支逻辑；即使关闭，仍会执行速度限幅：
  - `max_step = command_max_velocity_rad_s * dt`
  - `delta = clamp(target - prev, -max_step, +max_step)`
- 当前配置 `command_max_velocity_rad_s: 1.0`，若控制周期约 20ms，则单周期仅约 0.02rad，较大目标变化会表现为明显“慢慢追”。
- 当前配置还开启了 `enable_startup_smoothing: true`（3.0s）和 `enable_mode_rising_smoothing: true`，在上电阶段/模式切换后会进一步平滑目标。
- launch 默认 `gains_file` 来自 `FindPackageShare("trotbot_can_bridge")/config/control_gains.yaml`，若非 `--symlink-install` 或未重建，可能读取 install 目录中的旧 YAML，而非 src 中最新修改。

#### 7) 修复与验证

- 修复内容：
  - 核验运行态参数实际值（`enable_feedback_step_limit`、`command_max_velocity_rad_s`、`enable_startup_smoothing`、`startup_smoothing_duration_s`）。
  - 统一参数生效链路：显式 `gains_file` 指向工作区 src 配置，或重建安装后再启动。
  - 根据实机跟随性调大 `command_max_velocity_rad_s`（建议先试 6~12 rad/s 区间）。
- 验证命令/方法：
  - `ros2 param get /motor_protocol_node enable_feedback_step_limit`
  - `ros2 param get /motor_protocol_node command_max_velocity_rad_s`
  - `ros2 param get /motor_protocol_node enable_startup_smoothing`
  - `ros2 launch trotbot_can_bridge can_bridge.launch.py gains_file:=<abs_path_to_src>/control_gains.yaml`
- 验证结果：
  - 待你现场按上述步骤复测后回填。

#### 8) 结论

- 初步判断为“多重平滑/限幅叠加 + 参数文件来源不确定”导致的表象，不是单一 `feedback_step_limit` 开关失效。

### ISSUE-0002 前腿卡顿与前后腿发送频率长期失衡

- 发现时间：2026-04-27（夜间联调）
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`、`trotbot_can_bridge/can_transport_node`、`trotbot/config/champ/gait_minidog.yaml`、CAN overlay
- 优先级：P0 / P1 / P2
- 当前状态：In Progress（2026-04-29：DDS QoS + 每电机 200Hz 限频已实机对齐；卡顿需带载步态再确认后转 Closed）

#### 1) 现象

- 悬空跑时前腿明显“卡顿/顿挫”，后腿相对平滑。
- 实测前后腿发包频率长期不均衡，前腿显著低于后腿。
- 单独屏蔽后腿仅发前腿时，前腿观感明显改善。

#### 2) 复现步骤

1. 正常启动 launch，双总线都开启（`tx_enable_can0=true`，`tx_enable_can1=true`）。
2. 手柄持续前进，抓取 `/can_frames_tx_line` 与 `/rosout`（`CAN bus counts` / `TX enqueue bus split`）。
3. 统计每电机 10s 发送频率与 can0/can1 总线发包比。

#### 3) 期望与实际

- 期望：双总线打开时，前后腿发包总量接近 1:1；每电机频率接近同一量级。
- 实际：多轮测到 `can0:can1` 约 `1:2`，并出现个别电机极低频（早期常见 11/12/13，后续漂移到 22/23/51）。

#### 4) 影响评估

- 直接影响前腿动作连续性与步态跟随观感，导致“卡顿”。
- 在动态阶段造成关节更新不一致，放大姿态抖动和相位误差。

#### 5) 临时绕过

- 临时仅发前腿（`tx_enable_can0=true`，`tx_enable_can1=false`）可显著改善前腿观感。
- 临时仅发后腿可复现实验对照（确认问题并非单腿硬件缺陷）。

#### 6) 根因分析（当前结论）

- 不是单纯 CAN 链路错误：`TX bus health(5s)` 长时间 `fail/retry/drop=0`。
- 不是 overlay 缺失：已确认 CAN 插件正常加载，接口稳定 up。
- 核心问题是“发布侧到发送侧存在分布性失衡”，表现为不同电机命中频率长期不一致。
- `/joint_group_effort_controller/joint_trajectory` 维度完整（12关节），但下游每电机发包仍出现漂移型低频点，说明仅靠参数调节难以彻底消除。

#### 7) 已做修复/实验（按时间顺序）

- 增加观测：
  - `Command clamp stats(total)`（CHAMP/MIT限位裁剪率）
  - `TX bus health(5s)`（按总线 fail/retry/drop）
  - `TX enqueue bus split(5s)`（按总线入队分布）
- `SmoothJointCommand` 修复：
  - 从全局 `dt` 改为每关节独立 `dt`，显著改善“命令被压扁”问题。
- 启动平滑优化：
  - 增加 `limit_velocity_only_during_smoothing`，避免稳态持续压幅。
- 临时总线开关：
  - 新增 `tx_enable_can0/tx_enable_can1`，用于前后腿独立 A/B。
- 发布路径调整：
  - `OnTrajectory` 改为固定索引映射全覆盖发送（并对缺失位使用上次目标兜底）。
- 发送频率约束：
  - 新增 `enable_min_tx_refresh/min_tx_refresh_interval_s`
  - 新增 `enable_max_tx_rate_limit/max_tx_rate_per_motor_hz`
  - 结论：可改变分布形态，但尚未彻底消除“漂移型低频电机”。
- CAN 时钟核对与对齐官方建议：
  - 实机比特率为 1M；
  - overlay `assigned-clock-rates` 已由 200MHz 改为 300MHz（官方 1M 推荐值），重启后生效。
- 新增 200Hz 外部基线脚本（2026-04-29）：
  - `src/trotbot_can_bridge/scripts/el05_mit_stream_200hz_12_direct.sh`：按 12 电机固定 MIT 帧循环发送（默认 `TARGET_HZ=200`，`kp=20`）；
  - 用于先排除“上层控制链路调度误差”，直接对照 `cansend` 发送频率与电机反馈频率是否同量级。
- 新增协议层发送原因窗口统计（2026-04-29）：
  - `motor_protocol_node` 增加 `MP TX window(5s)` 日志，统计：
    - 轨迹触发发送 `tx_traj`
    - 最小刷新补发 `tx_refresh`
    - 最大频率限流跳过 `skip_max_rate`
    - 总线开关跳过 `skip_bus_disabled`
  - 目的是把“掉频”拆分到具体环节，避免仅看总线计数。

#### 8) 关键实测结论（摘录）

- 单前腿发送时：总线约 `560Hz`，观感明显改善。
- 单后腿发送时：总线约 `560Hz`，形态与单前腿镜像。
- 双腿同时发送时：多轮出现 `can0` 明显低于 `can1`，且个别电机频率异常低。
- 最近一次（CAN时钟300M + 双腿开启）虽总量接近 1:1，但仍有个别电机掉频（如 23、51），问题未彻底闭环。
- 2026-04-29 压测补充（新统计口径）：
  - 条件：`JointTrajectory` 外部压测目标 `200Hz`（`ros2 topic pub -r 200`）；
  - 实测：`motor_protocol_node` 5 秒窗口 `traj_cb=614`（约 `122.8Hz` 输入）；
  - 协议层发送：`tx_total=1848`，其中 `tx_traj=1776`、`tx_refresh=72`；
  - 限流跳过：`skip_max_rate=5592`（受 `max_tx_rate_per_motor_hz=60` 影响显著）；
  - 结论：当前“200Hz 目标掉频”主要受上游实际输入频率 + 协议层限频策略共同影响，不是单一 can0/can1 发送器失效。
- 2026-04-29 真链路三联采集（`trotbot_basic.launch.py use_can_bridge:=true rviz:=false`）：
  - `/joint_group_effort_controller/joint_trajectory`：`~200Hz`（窗口 1605 样本，均值 `200.03Hz`）；
  - `/can_tx_frames`：`~253.46Hz`（窗口 2038）；
  - `/can_rx_frames`：`~272.73Hz`（窗口 2192）；
  - `motor_protocol_node`：`MP TX window(5s): traj_cb=1000, tx_total=3000, tx_traj=2976~3000, tx_refresh=0~24, skip_max_rate=9000+`；
  - `can_transport_node`：`TX enqueue bus split(5s): enq_can0=342~495, enq_can1=1248~1334 (ratio=0.26~0.37)`；
  - 关键发现：`motor_protocol` 统计近似前后腿 1:1，但 `can_transport` 入队统计明显偏后腿（can1）；该“分层统计不一致”需优先排查 transport 分流判定/统计口径。
- **2026-04-29 根因与修复（已实机验证）**：
  - **根因**：`/can_tx_frames` 原使用 `SensorDataQoS()`（BEST_EFFORT + 浅队列），在高频连续 `publish`（12 电机 × 200Hz 量级）下 **DDS 侧丢帧**，`can_transport_node` 实际收到的帧数显著少于 `motor_protocol_node` 发布数；残存样本在统计上可出现 **can0/can1 比例失真**（此前观测 `enq_can1` 远大于 `enq_can0`）。
  - **修复**：
    - `motor_protocol_node`：`/can_tx_frames` 改为 **Reliable + `KeepLast(can_tx_frames_qos_depth)`**（默认 4000，参数见 `control_gains.yaml`）。
    - `can_transport_node`：订阅同一话题使用 **相同 QoS**（参数见 `bridge.yaml`）。
    - **对齐 CHAMP 200Hz**：`max_tx_rate_per_motor_hz` 默认由 `60` 调整为 **`200`**，避免每电机被错误削峰到 60Hz。
  - **验证**：同一 5s 窗口内 rosout 中 `TX enqueue bus split` 的 `enq_can0≈enq_can1`（ratio≈1.01–1.02），`enq` 总量与 `MP TX window` 的 `tx_total` 同量级，前后腿计数一致。
  - **验证抓包（联调机单次归档）**：启动 `trotbot_basic.launch.py use_can_bridge:=true rviz:=false gait_config_file:=gait_minidog.yaml`，另终端 `ros2 topic pub -r 20 /cmd_vel geometry_msgs/msg/Twist "{linear:{x:0.12,y:0,z:0},angular:{x:0,y:0,z:0}}"` 约 20s+，标准输出重定向至 `/tmp/verify_issue002.log`（路径仅作记录，可按需改）。摘录（示例）：
    - `TX enqueue bus split(5s): enq_can0=3485 enq_can1=3425 (ratio=1.018)`；
    - `MP TX window(5s): traj_cb=1000 ... tx_traj=6874(can0=3467 can1=3407) ...`；
    - `TX stats(5s): enq=6910 ok=6910 fail=0 ... drop_overflow=0`（`enq` 与 `tx_total` 同窗可对齐）；
    - `TX bus health(5s): can0/can1` 各项 `fail=0`，无溢出与异常丢弃。

##### 此次修复摘要（逻辑层面改了什么）

此次**未改**电机协议编解码、`DogMapper` 前后腿路由、`can_transport`「按 `frame.bus` 入队」的核心算法；修复集中在 **ROS 2 话题 QoS + 每电机发送上限参数**，消除「发布数 ≫ 订阅收到数」及由此导致的 **假 bus 占比**。

| 改动点 | 文件 / 节点 | 作用 |
|--------|-------------|------|
| **`/can_tx_frames` 发布 QoS** | `motor_protocol_node.cpp`、`control_gains.yaml` 参数 `can_tx_frames_qos_depth` | 由默认 **SensorDataQoS（BEST_EFFORT + 浅深度）** 改为 **Reliable + KeepLast(默认 4000)**，避免高频连续下发时 **DDS 层丢消息**。 |
| **`/can_tx_frames` 订阅 QoS** | `can_transport_node.cpp`、`bridge.yaml` 同上参数 | **与发布端 QoS 匹配**（Reliable + 同深度）；否则仍可能协商失败或丢帧。 |
| **每电机最大发送频率** | `control_gains.yaml`：`max_tx_rate_per_motor_hz` **60 → 200** | 与 CHAMP **`loop_rate: 200Hz`** 对齐；过低时每关节被压在 ~60Hz，产生大量 `skip_max_rate`，有效指令更新偏慢（与「失衡统计」叠加时更难排查）。 |

**上机使能后建议**：先看 rosout 中 `TX enqueue bus split` 是否仍接近 **1:1**，`TX stats` 的 `enq` 与 `MP TX window` 的 `tx_total` 是否同量级；再感受步态。若仍顿，优先记录是否有个别 `motor_id` 长期低频（再查限幅/平滑/地面接触等非 DDS 因素）。

#### 9) 下一步计划（下次联调入口）

- 继续做“每电机发送原因计数”诊断（按电机统计 skip 原因）：
  - bus 开关跳过
  - 最大频率限制跳过
  - 最小刷新补发命中
  - 真实发送成功计数
- 在不影响动作的前提下，先锁死“每周期每电机必发”路径做对照基线，再逐步恢复优化器（平滑/刷新/限速）找最小可用集合。
- 本轮先执行“200Hz 显式脚本 + candump 时间戳”对照：
  - 若发送与反馈接近 200Hz，则优先回查 `motor_protocol_node` 的调度/限频链路；
  - 若反馈显著低于发送，则优先回查电机上报参数、总线负载与驱动队列瓶颈。

#### 10) 结论

- 本问题已从“体感异常”收敛到“可量化的每电机发送分布异常”；
- **统计层面**：根因为 **`/can_tx_frames` DDS QoS 不当导致丢帧**，叠加 **`max_tx_rate_per_motor_hz=60` 与 200Hz 轨迹不匹配**；已通过 Reliable 深队列 + 200Hz/电机限频修正，**MP 与 transport 计数、can0/can1 占比已对齐（见上文验证摘录）**。
- **体感层面**：抬狗使能后仍需现场确认；若仍卡顿，再按每电机/地面工况分项排查（不限于本 ISSUE 原统计口径）。

### ISSUE-0003 `zero_sta(0x7029)` 导致断电后负向小偏移上电“绕圈回零”

- 发现时间：2026-04-30
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`EL05 参数配置`、`src/trotbot_can_bridge/scripts/el05_motor_cansend.sh`、上电/零位流程
- 优先级：P0 / P1 / P2
- 当前状态：In Progress

#### 1) 现象

- 狗子趴下断电后，若某关节被人工轻微负向拨动（例如 -10°），电机上电后按 `0~2π` 域解释为接近 `350°`；
- 若不重新 set_zero，控制回 `0°` 时会出现接近一圈的大幅旋转，不符合预期；
- 正向小偏移（例如 +10°）时表现正常，问题主要在“负向跨 0”场景。

#### 2) 复现步骤

1. 电机参数 `zero_sta=0`（默认）；
2. 狗子趴姿断电，人工将某关节轻微负向拨动；
3. 上电后直接执行回零/站立流程（不重设零位），观察关节是否出现大幅绕行。

#### 3) 期望与实际

- 期望：断电后小角度正负扰动都应在上电后被连续、就近解释，不发生绕圈回零。
- 实际：负向小偏移可能按 `0~2π` 域映射到接近 `2π`，触发大角度回零。

#### 4) 影响评估

- 上电阶段存在突发大角度转动风险，影响机械安全与联调可重复性；
- 易在“趴姿断电 + 人工挪动关节 + 未重置零位”的日常工况触发。

#### 5) 临时绕过

- 每次上电前统一 `set_zero`，避免沿历史绝对角回零（操作成本较高）；
- 或限制关节断电后人为拨动（不可完全依赖）。

#### 6) 根因分析

- EL05 手册写明 `zero_sta` 决定上电位置域：
  - `zero_sta=0`：`0~2π`
  - `zero_sta=1`：`-π~π`
- 当前默认 `zero_sta=0` 时，负向微小偏移会在上电解释中跨边界，导致“就近角”失真。

#### 7) 修复与验证

- 修复内容：
  - 在 `el05_motor_cansend.sh` 新增 `zero_sta` 相关命令：
    - `zero_sta_read`（通信类型17，读取 `0x7029`）
    - `zero_sta_set`（通信类型18，写 `0x7029`，由 `ZERO_STA_TARGET` 指定）
    - `zero_sta_save`（通信类型22，保存参数）
    - `zero_sta_apply`（一键：set -> save -> read）
  - 目标默认值建议：`ZERO_STA_TARGET=1`（上电域 `-π~π`）。
- 验证命令/方法：
  - `ZERO_STA_TARGET=1 bash src/trotbot_can_bridge/scripts/el05_motor_cansend.sh zero_sta_apply`
  - 复测“趴姿断电后正/负小偏移”两种情况，确认上电回零路径是否就近。
- 验证结果：
  - 已执行：
    - 单电机 `ID23`：`zero_sta_set(1)` + `zero_sta_save` 成功；
    - 全 12 电机：`zero_sta_set(1)` + `zero_sta_save` 成功；
    - 修正读取脚本后，`zero_sta_read` 可稳定回读类型17应答（先监听再发 + 过滤 tx-echo）：
      - 12 个电机均返回 `param=0x7029 value_u8=1`；
      - 示例应答：`110017FD [8] 29 70 00 00 01 00 00 00`（ID23）。
  - 结论：`0x7029(zero_sta)` 已批量写入并保存为 `1`；后续通过“断电后正/负小拨动再上电”做行为复测闭环。

#### 8) 结论

- 该问题与 `zero_sta` 参数边界定义直接相关，优先建议切到 `zero_sta=1` 并保存；
- 后续可考虑将上电流程加入“zero_sta 自检告警”，避免参数回退后再次触发。

### ISSUE-0004 启动时出现“瞬时站立脉冲”后再回落并缓慢站立

- 发现时间：2026-04-30
- 发现人：blue.ge
- 版本/分支：main（工作区联调）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`、启动编排流程、`champ_teleop` 按键触发
- 优先级：P0 / P1 / P2
- 当前状态：In Progress

#### 1) 现象

- 狗子初始趴姿，启动 launch 后应“平滑站立”；
- 实测会先出现一次“瞬时受力/近似瞬间站立”的脉冲，再回落到接近趴姿，随后才慢慢站起。

#### 2) 复现步骤

1. 电机已使能，狗子保持趴姿；
2. 启动完整控制链路（含 CHAMP + CAN bridge）；
3. 观察启动前 1~3 秒姿态变化。

#### 3) 期望与实际

- 期望：启动过程单调平滑，不出现突发脉冲抬升。
- 实际：启动初段插入一次明显高幅抬升，再回落并二次平滑站立。

#### 4) 影响评估

- 存在启动冲击风险，影响机械安全与体感；
- 会降低“遥控按键开机”流程可预测性与一致性。

#### 5) 临时绕过

- 仅在机械支撑充分时再启动；
- 人工放慢姿态变化并避免启动瞬间给速度命令。

#### 6) 根因分析（当前判断）

- 启动阶段缺少统一状态机门禁，`joint_trajectory` 首波目标可直接进入发送链路；
- 上电使能后控制链路与姿态目标切换不同步，可能出现“历史/默认高幅目标”先命中一次；
- 当前流程“脚本能力 + 节点控制”混用，但无统一的 `Idle -> EnableInit -> SoftStand -> Running` 编排。

#### 7) 修复与验证

- 修复内容：
  - 新增节点：`trotbot_can_bridge/power_sequence_node`（状态含 `ProneHold` 等，见 `power_sequence_node.cpp`）；
  - 启动门禁：`motor_protocol_node` 新增 `enable_power_sequence_gate`，未收到 `/power_sequence/gate_open=true` 前不透传 `joint_trajectory`；
  - 上电流程默认跳过 `set_zero`（可选参数保留），执行 `Precheck -> EnableInit(使能+MIT首帧0，不开主动上报) -> SoftStand`；
  - 下电流程实现：`SoftProne -> DisableMotors -> Idle`；
  - 按键触发实现：`L1+R1` 或 `□` 长按上电/start，`L1+R1+Share` 长按下电，`○` 长按趴下至 `ProneHold`（与话题 `shutdown` 区分）。
- 验证命令/方法：
  - `ros2 launch trotbot_can_bridge can_bridge.launch.py`
  - 模拟长按（或实机遥控）触发三段流程，抓取 rosout 状态切换与门禁日志。
- 验证结果：
  - 状态机与门禁按预期生效（摘录）：
    - `power_sequence state -> Precheck -> EnableInit -> SoftStand -> Running`
    - `Power sequence gate changed: gate_open=1`
    - `power_sequence state -> SoftProne -> Disable -> Idle`
    - `Power sequence gate changed: gate_open=0`
  - 门禁关闭窗口 `MP gate window(5s): gate_open=0 skip_gate>0`，表明启动前轨迹突发被拦截；进入 `Running` 后恢复发送。

#### 8) 结论

- 问题本质是“启动编排缺失 + 门禁缺失”而非单个参数失配；
- 已通过状态机化启停流程闭环 RQ-007/RQ-008/RQ-013（软件实现完成），待实机体感回归确认后关闭问题单。

### ISSUE-0005 RViz RobotModelFeedback 腿叠在机身、`no transform from fb/...`

- 发现时间：2026-05-01
- 发现人：联调
- 版本/分支：main（工作区）
- 关联模块：`trotbot_can_bridge/motor_protocol_node`、`robot_state_publisher`（`frame_prefix:=fb/`）、`trotbot/rviz/trotbot.rviz`
- 优先级：P1
- 当前状态：Resolved

#### 1) 现象

- `fb/base_link` 正常，其余 `fb/lf1` 等报无变换或 RViz 反馈模型腿挤在身体中间。

#### 2) 复现步骤

1. `ros2 launch trotbot trotbot_basic.launch.py use_can_bridge:=true rviz:=true`
2. 配置中将 `enable_rx_decode_log` 设为 `false`（为减少日志/字符串反馈）
3. 打开 RViz 中 RobotModelFeedback

#### 3) 期望与实际

- 期望：`/joint_states_feedback` 持续更新，`robot_state_publisher_feedback` 发布完整 `fb/*` 动态 TF，反馈模型与实机关节一致。
- 实际：`OnRxFrame` 在 `enable_rx_decode_log=false` 时整函数提前返回，不解析 CAN、不调用 `PublishFeedbackJointStates()`，反馈 RSP 收不到关节状态，仅余静态 TF 与 `fb/base_link`。

#### 4) 影响评估

- RViz 双模型对比失效；易误判为 TF 前缀或 URDF 错误。

#### 5) 临时绕过

- 保持 `enable_rx_decode_log: true`。

#### 6) 根因分析

- 将“解码日志开关”误用作“是否处理 RX 帧”的总开关，与参数语义不符。

#### 7) 修复与验证

- 修复内容：
  - 始终执行 Unpack/DecodeFeedback 与反馈角更新；`enable_rx_decode_log` 仅控制解码字符串话题与 INFO 节流；`/joint_states_feedback` 发布端 QoS 为 `SensorDataQoS`。
  - **补充（2026-05-01）**：原先仅在收到匹配 CAN 反馈帧时发布 `JointState`；上电门禁未打开或总线无反馈时话题空闲，`robot_state_publisher_feedback` 无动态 TF，RViz 反馈模型仍异常。已增加 `feedback_joint_states_timer_hz`（默认 50）定时发布，与 CAN 解耦，便于桌面/未使能时仍保持 TF 树。
- 验证命令/方法：`enable_rx_decode_log:=false` 时 `ros2 topic hz /joint_states_feedback` 仍有速率；RViz 中 RobotModelFeedback 无整链 `no transform`。
- 验证结果：待实机/有 CAN 环境复测（逻辑与代码路径已修正）。

#### 8) 结论

- 根因为 `enable_rx_decode_log` 错误门控整条 RX 处理；修复后该参数仅影响日志侧载。
