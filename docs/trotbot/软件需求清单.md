# TrotBot 软件需求清单（按优先级）

> 本文档仅保留软件需求与实现状态，不含排期/人天/里程碑计划。  
> 状态定义：`[x] 已完成`（代码已实现且可按下方验收要点验证），`[ ] 未完成`（当前仓库未实现或未闭环）。  
> **关联文档**：实现细节与话题拓扑见 `docs/trotbot/架构说明.md`；运行与标定入口见 `docs/trotbot/本地运行指南.md`。

## 1. 范围与目标

- **范围**：`CHAMP -> CAN Bridge -> 电机反馈 -> RViz/状态话题`（含上/下电编排与安全相关参数）；**控制增强**含 MIT **前馈力矩**（标量/表查，见 **RQ-022**）与 **基于多体动力学的模型前馈**（**Pinocchio** 等，见 **RQ-025**）；**交付态人机提示**含 WS2812 **状态灯环**（见 **RQ-023**）；**运维标定**含维修后 **遥控触发机械零位**（见 **RQ-024**）。  
- **目标**：形成可逐条开发与验收的软件需求清单；**优先**安全与可控，**其次**可观测与易用性，**再次**架构演进（见 §6）。

## 2. 高优先级（P0）

- [x] **RQ-001 关节命名映射与顺序一致性**  
  - **说明**：桥接层按 CHAMP 关节名顺序映射 12 关节，与 URDF / `joint_trajectory` 顺序一致。  
  - **验收要点**：`JointTrajectory` 关节名顺序与 `kChampJointNames` 一致即可正确映射。  
  - **代码依据**：`trotbot_can_bridge/include/trotbot_can_bridge/dog_mapper.hpp` 中 `kChampJointNames`。

- [x] **RQ-002 关节到 CAN 总线与电机 ID 映射**  
  - **说明**：12 关节固定映射到 `11/12/13/21/22/23/51/52/53/61/62/63`，前腿 `can0`，后腿 `can1`。  
  - **验收要点**：配置文件与 `DogMapper` 对同一关节索引指向同一 `(bus, motor_id)`。  
  - **代码依据**：`dog_mapper.hpp`、`trotbot_can_bridge/config/motor_map.yaml`。

- [x] **RQ-003 URDF角 -> 电机角 正向转换**  
  - **说明**：按 `motor_pos_rad = sign * urdf_pos_rad + offset` 转换后下发 MIT 帧。  
  - **验收要点**：修改 `joint_signs` / `joint_offsets_rad` 后，同一 URDF 指令对应 CAN 侧位置线性变化符合公式。  
  - **代码依据**：`trotbot_can_bridge/src/motor_protocol_node.cpp` 的 `mapped_position` 计算。

- [x] **RQ-004 标定参数入口（sign/offset）**  
  - **说明**：支持 `joint_signs[12]`、`joint_offsets_rad[12]`（及镜像展开）参数化，可从 YAML 加载。  
  - **验收要点**：不改代码仅改配置即可切换标定；与 `calibration_profiles.md` 口径一致。  
  - **代码依据**：`motor_protocol_node.cpp` 参数声明；`trotbot_can_bridge/config/control_gains.yaml`。

- [x] **RQ-005 电机角 -> URDF角 反向转换并发布反馈关节状态**  
  - **说明**：把反馈机械角反解为 URDF 角并发布 `sensor_msgs/JointState`。  
  - **当前实现**：发布到 **`/joint_states_feedback`**（命名区别于指令链路的 `/joint_states`）；默认 **`feedback_joint_states_timer_hz`** 定时发布，与 CAN 反馈到达时刻解耦，保证 RViz `fb/` TF 在无反馈窗仍可刷新。**RX 侧同时接受 CAN ID 高 5 位为 `0x02`（应答反馈）与 `0x18`（通信类型 24 主动上报），8 字节数据域解析一致，任一到达即更新该电机缓存。**  
  - **验收要点**：RViz 双模型中反馈模型与 `/joint_states_feedback` 一致；断 CAN 后定时发布仍可维持上次位置（或文档约定的占位策略）。  
  - **代码依据**：`protocol_codec.hpp` `DecodeFeedback()`；`motor_protocol_node.cpp` 中 `champ_feedback = (current_angle - offset) / sign` 与 `PublishFeedbackJointStates()`；`control_gains.yaml` 中 `publish_feedback_joint_states`、`feedback_joint_states_topic`、`feedback_joint_states_timer_hz`；扩展 **`publish_feedback_velocity_effort`** / **`feedback_joint_fill_nan`** 见 **RQ-012**。  
  - **补充**：速度/力矩/离散状态位的结构化承载见 **RQ-012**（本条目仍以 **`position`** 为主验收 RViz 反馈链）。

- [x] **RQ-006 机械限位保护（按 URDF 限位映射到电机限位）**  
  - **说明**：将 URDF 关节指令限幅（并可映射到电机域）后执行发送前截断保护。  
  - **验收要点**：指令超出配置区间时下发值被限制，日志或计数器可观测截断发生。  
  - **代码依据**：`motor_protocol_node.cpp` 的 `ClampJointCommand()`、`RebuildMotorLimitTable()`、`ClampMotorCommand()`；`control_gains.yaml` 的 `joint_cmd_min_rad/max_rad`。

- [x] **RQ-007 上电初始化流程编排（安全启动）**  
  - **说明**：可选零位 / 前置校验 → 使能与 MIT 首帧 → 启动门禁 → 软站立等顺序由 `power_sequence_node` 编排；`motor_protocol_node` 可通过门禁拦截轨迹。  
  - **当前实现补充**：Launch 阶段可选 **`enable_active_report_at_launch`** 提前打开周期上报（`0x18`），便于无 **`start`** 时 RViz 仍有反馈；与「EnableInit 内完整使能链」可重复发报文，属预期。**默认机型启动**：`trotbot_basic.launch.py` / `champ_controllers.launch.py` 使用 **`description_file=minidog_champ.urdf.xacro`**、**`gait_config_file=gait_minidog.yaml`**，避免与旧款 URDF/腿长不一致；其它整机需显式覆盖参数。  
  - **验收要点**：未 `start` 时轨迹不透传（门禁关闭）；`start` 后进入可控运行态；上报与使能行为与 `power_sequence.yaml` 一致。  
  - **代码依据**：`power_sequence_node.cpp`、`power_sequence.yaml`；`motor_protocol_node` `enable_power_sequence_gate`；launch 默认参数。

- [x] **RQ-008 下电收拢流程编排（安全停止）**  
  - **说明**：平滑趴下 → 到位判定 → 全机关节失能 → 退出控制门禁；**趴下保持使能**（`ProneHold`）与**下电失能**（`Disable`）拆分。  
  - **验收要点**：`prone`/Circle 长按进入趴下保持；`shutdown`/L1+R1+Share 完成失能；`/power_sequence/command` 文本指令可触发同等路径。  
  - **当前状态**：已实现并接入遥控与 `/power_sequence/command`。  
  - **代码依据**：`power_sequence_node.cpp` 状态机与参数。

## 3. 中优先级（P1）

- [x] **RQ-009 MIT 控制参数化下发**

- [x] **RQ-010 镜像参数展开能力**  
  - **说明**：支持由 `hip/thigh/calf` 三参数展开 12 维偏移，减少重复配置。  
  - **代码依据**：`BuildMirrorJointOffsetsRad` 与 `expand_joint_offsets_from_mirror`。

- [x] **RQ-011 RViz 双模型（指令态 + 实时态）**  
  - **说明**：同屏显示指令模型与反馈模型（不同 TF 前缀/固定 Frame）。  
  - **验收要点**：`RobotModel` 与 `RobotModelFeedback` 同时显示；反馈侧订阅 `/joint_states_feedback`。  
  - **代码依据**：`trotbot/rviz/trotbot.rviz`；`trotbot_basic.launch.py` 中 `robot_state_publisher_feedback` 与 remap。

- [x] **RQ-012 反馈结构化发布（位置 / 速度 / 力矩 / 状态与故障位）**  
  - **说明**：在 **RQ-005** 已提供位置型 `JointState` 的基础上，将解码端已有量（见 `MotorFeedback`：`current_speed`、`current_torque`、`mode_status`、各类 error 位、`current_temp` 等）以 **非字符串、可订阅** 的形式输出，供算法、监控与记录复用，减少对 `/motor_feedback` 字符串的依赖。  
  - **当前实现**：`OnRxFrame` 按关节缓存 CHAMP 顺序下的 **`velocity`**（rad/s，`ω_champ = ω_motor / joint_signs[i]`，直连模型）与 **`effort`**（Nm，EL05 报告力矩，与 MIT `tau_ff` 同号约定，不经 `joint_signs` 二次取反）；`PublishFeedbackJointStates()` 在 **`publish_feedback_velocity_effort=true`** 时填充 `JointState.velocity`/`effort`，缺失样本由 **`feedback_joint_fill_nan`** 决定 NaN 或 0。可选 **`publish_motor_diagnostics`** 向 **`motor_diagnostics_topic`**（默认 `/diagnostics`）发布 **`diagnostic_msgs/DiagnosticArray`**：`mode_status`、`temp_c`、`fault_mask` 及各位 fault。  
  - **验收标准（DoD）**：  
    1. ~~至少一条稳定话题~~：`/joint_states_feedback` 扩展 **`velocity`/`effort`**，12 维与 CHAMP 顺序一致；语义见 `架构说明.md`。  
    2. ~~电机/模式状态字或离散故障位~~：由 **`DiagnosticArray`** KeyValue 承载（不解析 `/motor_feedback` 字符串）。  
    3. ~~文档与参数表~~：`declare_parameter` 与 `control_gains.yaml`、`架构说明.md` 一致。  
  - **代码依据**：`motor_protocol_node.cpp` `OnRxFrame`、`PublishFeedbackJointStates()`、`PublishMotorDiagnostics()`；参数 `publish_feedback_velocity_effort`、`feedback_joint_fill_nan`、`publish_motor_diagnostics`、`motor_diagnostics_topic`。  
  - **关联**：**RQ-022** 可使用 **`effort`** 做离线填表与在线对照。

- [x] **RQ-022 MIT 前馈力矩补偿（重力 / 步态一致性）**  
  - **背景**：各电机需**按关节区分**的力矩前馈以抵消重力等；CHAMP 轨迹主要为**位置**，桥接层在 MIT 帧写入 **`tau_ff`**。  
  - **当前实现**：**`enable_joint_tau_ff=false`**（默认）时行为与旧版一致：每总线仅 **`runtime_tau_can0/can1`**（来自 **`default_tau_ff`** 与 **`/mit_gains_cmd`**）。**`enable_joint_tau_ff=true`** 时：`tau_cmd = Clamp(runtime_tau_bus + gravity_ff_scale * tau_ff_nominal_nm[i], kTMin, kTMax)`，与轨迹及 **`OnTxRefreshTimer`** 刷新路径同步；**`tau_ff_nominal_nm`** 为 12 维，顺序同 **`DogMapper::kChampJointNames`**。  
  - **接口假设**：前馈表可由 **`MotorFeedback.current_torque`** / **RQ-012** 的 **`JointState.effort`** 离线统计填入 YAML。  
  - **安全**：默认 **`enable_joint_tau_ff=false`**；**`gravity_ff_scale`** 默认 **0.5**；符号与 **`tau_ff_nominal_nm`** 与 EL05/MIT 一致（不经 `joint_signs` 二次取反），限幅沿用协议边界。  
  - **验收标准（DoD）**：  
    1. YAML **`tau_ff_nominal_nm`** + **`enable_joint_tau_ff`** 可使各关节 MIT **`tau_ff`** 不全相等（配置允许时）。  
    2. 实测对比留现场记录口径。  
    3. **`control_gains.yaml`** 注释与 **`架构说明.md`** 已同步。  
  - **代码依据**：`motor_protocol_node.cpp` **`ComputeMitTorqueFf()`**、`SendMitFrame()`、`OnTxRefreshTimer()`；`control_gains.yaml` **`enable_joint_tau_ff`**、**`gravity_ff_scale`**、**`tau_ff_nominal_nm`**。

- [ ] **RQ-025 基于 Pinocchio（URDF）的关节前馈力矩 / 动力学能力扩展**  
  - **背景 / 用户场景**：CHAMP 规划主输出为**关节位置**；在重力与动力学作用下，**实际角与指令角**存在偏差，步态难与预期完全一致。**RQ-022** 已支持按关节 `tau_ff_nominal_nm` 的**标量/查表**前馈；本需求在**同一 MIT 桥接思路上**，引入 **Pinocchio** 对当前机体位形做**逆动力学 / 最小为 RNE 重力项**，得到**随姿态变化**的力矩前馈，经 **MIT `tau_ff`** 下发，以提升行走等动态过程表现，并保留后续复用 **Pinocchio 微分运动学、质心/动量、雅可比** 等能力以迭代运动性能。  
  - **与 RQ-022 关系**：**RQ-022** 为**配置化前馈**；**RQ-025** 为**模型在线（或同频）计算前馈**。二者可 **叠加或二选一**（需参数 `enable_pinocchio_tau_ff` 等明确定义优先级，避免双计重力）。  
  - **必须满足的行为（Happy path）**：  
    1. 使用与实机标定一致的 **URDF**（如 `minidog_champ.urdf.xacro` 链出模型）构建 Pinocchio 模型；关节顺序与 **CHAMP / `kChampJointNames`** 对齐。  
    2. 以**当前关节角**（优先 **`/joint_states_feedback` 的 `position`**，与 **指令** `joint_trajectory` 的融合策略在实现中二选一并文档化）计算至少 **RNE 重力项** 或等价的 **gravity(model,q)** 关节力矩，输出 **12 维**前馈，经控制周期发布。  
    3. 与 **`motor_protocol_node` 的 `ComputeMitTorqueFf()`** 集成：将 Pinocchio 输出作为 **每关节 `tau_ff` 的源之一**（或经话题由桥接节点订阅后写入内部缓存），与现有 **`enable_joint_tau_ff` / `gravity_ff_scale`** 安全策略一致。  
  - **推荐软件架构（基于当前代码迭代，非推翻）**：  
    - **方案 A（推荐首版）**：新增包 **`trotbot_dynamics`（或 `trotbot_pinocchio_node`）** —— 节点内 `find_package(pinocchio)` 加载 URDF，订阅 `joint_states_feedback`（及可选 `joint_trajectory` 点插值），以固定周期发布 **`std_msgs/Float64MultiArray` 或 `sensor_msgs/JointState`（仅 `name`+`effort`）** 到话题如 **`/model_tau_ff`**；`motor_protocol_node` 增加**可选订阅**，与查表前馈按策略合并。  
    - **方案 B**：在 **`motor_protocol_node` 内**直接链 Pinocchio（减少延迟、单进程，但依赖与测试范围变大）。  
    - **浮基座 vs 定基座**：若 URDF 为**单刚体基座 + 腿**，且 CHAMP 不在世界系估计基座，**首版**可采用**定基、仅腿关节**的 RNE 重力补偿；若需全机浮动基动力学，需与 **state_estimation** 及基座位姿估计衔接，列为**二期**。  
  - **力矩符号与「电机 mapping」**：Pinocchio 输出为 **URDF/关节空间**力矩，方向与 **EL05 报告力矩、MIT `tau_ff`** 的约定须**单独标定**；**不可**默认沿用位置 `joint_signs` 的同一套取反（**RQ-012** 中 `effort` 与 `sign` 关系已独立说明）。实现上在**单点函数**内完成 **URDF τ → 电机 τ_cmd** 的校核，并附**台架对拉**或**静力平衡**记录。  
  - **安全与失败场景**：  
    - 与 **RQ-022** 相同类风险：方向反了会**加重偏差** → 默认 **总增益 < 1**、**可一键关闭**、**限幅** `kTMin/kTMax`；异常（URDF 未加载、q 非有限值、计算超时）时**回落**为无模型前馈。  
    - 在现网 **CHAMP + motor_protocol** 上**增量**交付，不替代上/下电与门禁。  
  - **ROS 2 / 生态调研说明（非「现成四足整包」承诺）**：  
    - **Humble** 下可 `apt` 安装 **`ros-humble-pinocchio`**，在自研节点中直接链接（与 [Pinocchio 官方文档 / index.ros.org](https://index.ros.org/p/pinocchio/) 一致）。  
    - **`kinematics_interface_pinocchio`** 主要服务 **ros2_control** 的**运动学接口**替换 KDL，**不**直接提供「四足步态 + MIT 力矩链」的即用节点。  
    - 高阶**任务空间**控制常见 **TSID、Crocoddyl、OCS2** 等（常与 MPC 联用），集成成本高于「单节点 RNE 重力前馈」；本需求首阶段以 **Pinocchio 单体 + 自研 ROS 接口** 为主，**不依赖**「开箱即用机器狗专用 ROS 包」。  
  - **验收标准（DoD）**：  
    1. 文档化：**节点图、话题名、输入输出频率、与 `motor_protocol_node` 合并逻辑**。  
    2. 实机或仿真：**开启模型前馈且增益从小到大**，站立/慢走对比记录（姿态误差或关节误差趋势优于「关模型前馈」基线之一即可）。  
    3. **`架构说明.md`** 增补 Pinocchio 数据流；**`package.xml`/`CMakeLists.txt`** 依赖可追溯。  
  - **代码依据（接入点）**：`motor_protocol_node.cpp` **`ComputeMitTorqueFf()`**、`SendMitFrame()`；关节顺序 **`DogMapper::kChampJointNames`**；URDF **`description_file`** launch 参数。

## 4. 低优先级（P2，易用性与运维）

- [x] **RQ-013 手柄触发系统级上/下电动作（PS4）**

- [x] **RQ-023 WS2812 圆环状态灯（无屏交付 / 开机就绪提示）**  
  - **背景 / 用户场景**：研发阶段可在桌面通过 UI 启动 launch；**交付客户**时常无显示器，需凭**灯语**判断「开机是否完成、栈是否就绪、是否可操作遥控」。期望鲁班猫 **RK3588（LubanCat RK3588x 系列）** 通过 **40PIN 引出 GPIO** 驱动 **WS2812（2812）圆环灯带共 16 颗灯珠**，以**颜色/静态图案**区分整机状态（避免高频闪烁干扰辨识时，默认以**稳态灯效**为主，可选低速呼吸等）。  
  - **硬件接口（已选线）**：**40PIN 物理引脚 Pin 16** 接 **WS2812 数据线（DIN）**。针对 **LubanCat-4-V1**，Pin16 与 SoC 侧 **GPIO 冻结映射为 `GPIO3_C1`**（详见仓库附图 **`docs/image/LubanCat-4-V1 pins.png`**）。该脚为 **GPIO**；实现时用 **`gpioinfo`/`gpioset`** 等在对应 **`gpiochip` 与 line** 上操作（不同镜像/DTS 下 **line 编号可能不同**，以板上实测为准）。**灯珠数量 16**（`led_count` 等参数默认 16，顺序、亮度上限可配）。**5V 供电、地与信号共地**以硬件 BOM/接线说明为准；电平若需 3.3V↔5V 需按实际模块处理。  
  - **说明**：建立 **灯带状态 ← 机器人状态** 的映射表（可与 `power_sequence_node` 的 `/power_sequence/state`、`/power_sequence/gate_open` 及关键节点存活信息组合）；例如：**系统已拉起且允许控制（如 Running + gate 语义就绪）→ 绿灯常亮/环亮**，表示「可开始遥控」；其它状态如初始化中、故障、失能等用不同颜色或分段显示。**独立小型状态机**（可在单节点内实现）订阅已有话题，输出 **16 像素** RGB 帧，更新频率与 WS2812 时序匹配即可。  
  - **必须满足的行为（Happy path）**：  
    1. 在 RK3588 上经 **Pin16 → DIN** 能稳定驱动 **16 颗 WS2812**（顺序、亮度上限可配置）。  
    2. **GPIO / 用户态访问**遵循官方说明：采用 **Linux GPIO 子系统**，优先 **libgpiod**（`gpiodetect` / `gpioinfo` / `gpioset` 等）。**Rockchip 命名** `GPIO3_C1` 含义见野火文档（控制器 **3**、端口 **C**、索引 **1**）。**官方参考**：[野火鲁班猫 RK3588 — GPIO 控制](https://doc.embedfire.com/linux/rk3588/quick_start/zh/latest/quick_start/40pin/gpio/gpio.html)（含命名规则、sysfs **legacy**、**FAQ**：**`Device or resource busy`** 多为设备树已占用）。实现时在代码/YAML 中固化 **`GPIO3_C1`**，并用 **`gpioinfo <chip>`** 查到该脚在当前系统上的 **`gpiochip` + line**，勿硬编码 sysfs 全局编号（内核版本间可能变化）。  
    3. **优先**调研 **ROS 2 Humble** 下可移植的 **WS2812 / NeoPixel** 类节点或库（**bit-bang / SPI 等**需满足 800kHz 级时序时，以实机测试为准）；否则在 **可独占的 `gpiod` line** 上封一层用户态发码，再包成 ROS 2 节点。  
    4. 将「狗子状态」映射为「灯效配置」（YAML：状态字符串 → 颜色/图案），无屏场景下用户可根据文档读懂灯语。  
    5. **开机自启动**：与整机 launch/systemd 编排衔接（具体单元文件路径在实现阶段写入 `README` / 运行指南）。  
  - **安全与失败场景**：  
    - **硬件不可用**：接线错误、GPIO 被占用、无法产出可靠 WS2812 比特流 → **不得阻塞**运动栈启动；灯带节点启动失败应 **降级为仅日志告警**，整机仍以「无灯语」模式运行。  
    - **软件未自启**：若 systemd **未**拉起机器人或灯带 launch → 用户看不到绿灯并非「已就绪」，文档需明确：**绿灯语义仅在完整栈成功启动且映射状态满足时成立**。  
    - **电气安全**：灯带供电与信号地与整机共地方案由硬件确认；软件侧限制最大亮度参数以防过热（可选）。  
  - **接口假设**：**Pin16 / `GPIO3_C1`** 在所用镜像中可作为 **GPIO 输出**独占使用；ROS 包需验证 **Humble + aarch64**。  
  - **关联**：`/power_sequence/state`（`Idle` / `Running` / `Fault` 等）为灯语主输入之一；可选叠加 **节点存活 / CAN 连通** 作为「就绪」条件。  
  - **验收标准（DoD）**：  
    1. 实机无显示器：完成从开机到「遥控允许」的全链路时，**16 灯**环进入文档定义的 **就绪态灯效**（示例：绿灯）。  
    2. 灯效与状态对照表写入仓库（Markdown 或 YAML 注释），并写明 **LubanCat-4-V1：Pin16 ↔ `GPIO3_C1`、`led_count=16`**（及所用镜像下 **`gpiochip`+line** 实测记录）。  
    3. 灯带节点异常退出时，控制栈行为不受影响；相关 launch/README 记录 **降级策略** 与 **自启动依赖**。  
  - **代码依据**：包 **`trotbot_status_led`**：`status_led_node` 订阅 `/power_sequence/state`、`/power_sequence/gate_open`；默认映射 **`config/status_led_map.yaml`**，参数 **`config/status_led_params.yaml`**（**`ws2812_backend`**：`spidev` / `rockchip_mmap` / `gpiod`）；launch **`trotbot_status_led/launch/status_led.launch.py`**；整机可选 **`trotbot_basic.launch.py`** 参数 **`use_status_led`**（默认 **false**）。**SPI 发码**见 **`src/trotbot_status_led/docs/SPI_WS2812_RK3588.md`**；**GPIO mmap** 实测模板：**`src/trotbot_status_led/docs/RK3588_GPIO_WS2812_HAL.md`**。引脚附图 **`docs/image/LubanCat-4-V1 pins.png`**；**DIN** 可 **Pin19（SPI0 MOSI）** 或 **Pin16（GPIO3_C1）**，与参数一致即可。

- [x] **RQ-024 维修后遥控触发十二电机机械零位（set_zero）**  
  - **背景 / 用户场景**：结构断裂修复或重装后，关节机械零位可能漂移，需**偶尔**重新执行电机侧 **set_zero**（与 MIT 侧 URDF 标定 `joint_offsets` 不同层次）。日常启动为安全起见默认**不在启动序列中自动零位**（见配置项）。  
  - **说明**：在 **launch 已运行**、用户**尚未执行 `start` 上电流程**（整机仍处于可认为是「未使能 / 未进入 Running」的安全窗口）时，通过手柄 **Option（选项）键** **长按**（默认 **`set_zero_longpress_s`**）触发 **12 路 EL05 机械零位指令**，效果等价于现行手动脚本对每轴下发的 set_zero 帧；亦可 **`ros2 topic pub ... /power_sequence/command data:=set_zero`**（无手柄调试）。  
  - **必须满足的行为（Happy path）**：  
    1. 推荐与 **`send_set_zero_in_startup: false`** 的日常口径配合使用；遥控路径与启动自动零位无硬互斥，但避免二者逻辑混淆。  
    2. 触发键：**PS4 Options**，索引 **`button_option`**（默认 9，**务必 `ros2 topic echo /joy` 实测**）；**`-1`** 关闭手柄触发、仅保留话题命令。  
    3. 可观测日志 **`WARN`**（成功下发）；遥控/话题触发时在 **`TryRemoteSetZero`** 内按 **`set_zero_repeat_count`**（默认 3）连续发送完整 **`SendSetZeroAll()`**，轮间隔 **`set_zero_repeat_interval_ms`**（默认 40 ms）；可选 **`set_zero_event_topic`** 发布 `set_zero_ok:…` / `set_zero_rejected:…`。  
  - **安全与失败场景**：  
    - **硬门控**：仅 **`state==Idle`** 且 **`/power_sequence/gate_open==false`** 时允许；否则 **WARN（节流）** 拒绝。**首期未接 `mode_status`**（若后续加固可订阅电机反馈）。  
    - 误触：**长按 ≥ `set_zero_longpress_s`（默认 2 s）**；松开 Option 后可再次触发。  
    - 与 **`el05_motor_explicit_cmds_12_direct.sh`**、其它 **cansend** 脚本勿并发抢总线。  
  - **接口假设**：依赖 `/joy`、`SendSetZeroAll()` → `/can_tx_frames`。  
  - **与现有配置/脚本关系**：  
    - **`send_set_zero_in_startup`**：`power_sequence.yaml`。  
    - 脚本 §ZERO 与 **`SendSetZeroAll()`** 同源：**CMD 0x06**、`data[0]=0x01`、全 ID 映射见 `DogMapper`。  
  - **验收标准（DoD）**：见 §5 电机零位语义；非 Idle 拒绝并有日志。  
  - **代码依据**：`power_sequence_node.cpp`：`TryRemoteSetZero()`（内嵌多轮 **`SendSetZeroAll()`**），`enable_remote_set_zero` / `button_option` / `set_zero_longpress_s` / `set_zero_repeat_count` / `set_zero_repeat_interval_ms` / `set_zero_event_topic`；`power_sequence.yaml` 同上。
